# 方向发现与用户确认门

## 1. 目的

方向发现门防止团队把会改变市场、产品、技术、财务和合规结论的缺失信息当作普通假设。它位于 Project Brief 和领域任务 DAG 之前，只负责澄清与记录用户决策，不产出领域结论。

## 2. 触发与跳过

以下情况触发：

- 新创业项目、BP、从 0 到 1 产品或重大工作流仍缺少核心方向；
- 输入存在互相冲突的市场、用户、产品或约束；
- 用户要求推荐方向，但没有授权团队直接定案；
- 两个以上可行选择会显著改变成本、合规或交付物。

以下情况跳过：

- 用户已明确问题、用户、产品 / 服务形态和首要市场；跨境任务还已明确供应来源与首要渠道 / 履约模式；
- 机械编辑、格式调整、明确验收的局部修改；
- 只缺品牌名、排版等低影响偏好，可披露假设后继续。

## 3. 交互协议

1. 先从用户输入与项目上下文提取已知字段，不重复询问。
2. 每个需要用户回复的回合只问一个关键问题。
3. 用户没有方向时提供 2–3 个真实可行的方向包，使用同一组判据比较。
4. 可以给条件式建议，但不得自动选择。
5. 用户明确选择后生成 `confirmed` 方向简报。
6. 用户明确授权团队代选且说明无需再次确认时，生成 `delegated` 方向简报；授权范围外的字段仍需询问。
7. 沉默、超时、“随便”或“你看着办”不能自动变成完整代选授权。

方向包使用统一结构：

| 字段 | 内容 |
|---|---|
| 方向 | 中性名称与一句话定义 |
| 目标用户与问题 | 谁在什么场景遇到什么问题 |
| 产品 / 服务形态 | 计划提供什么 |
| 首要市场与渠道 | 先在哪里、通过什么路径验证 |
| 核心取舍 | 主要收益、代价与约束 |
| 待验证假设 | 后续必须用证据关闭的事项 |

## 4. 状态与放行

```text
intake → ambiguity-scan
       ├─ clear → confirmed
       └─ material-gap → clarifying → options-presented
          → awaiting-confirmation → confirmed | delegated
confirmed | delegated → project-brief-freeze → dag-ready
```

只有 `direction-brief.dag_entry_authorized == true` 才能启动领域 DAG。等待用户输入不计入任务返工次数，也不能因等待时间过长自动选择默认方向。

## 5. 防回归场景

| 输入 | 期望行为 |
|---|---|
| “写一份完整的跨境电商BP” | 阻塞领域DAG；询问具体品类 / 服务方向，或提供方向包等待选择。 |
| “产品确定，但不知道做美国还是欧洲” | 只澄清会改变市场选择的首要约束，比较市场后等待确认。 |
| “给三个方向，你们选最好的并直接做，无需再确认” | 记录完整代选授权、展示比较与选择依据后进入DAG。 |
| “中国供应链、美国租房者、轻量收纳、Amazon+Shopify，写BP” | 原请求即确认，不额外提问。 |
| “把现有BP标题改名并修目录” | 机械任务，跳过方向发现。 |

关键断言：

```text
pending_user_questions <= 1
selected_direction requires selection_authority in [user, bounded, full]
bp_dag_started implies dag_entry_authorized == true
dag_entry_authorized implies user_confirmation_evidence is not empty
```

## 6. 失败与退出

- 信息不足：`needs-direction`，等待用户，不算失败。
- 用户拒绝全部方案：回到 `clarifying`，下一轮仍只问一个问题。
- 两轮仍无法确认：允许用户选择方向、明确授权代选、只获取占位模板或终止。
- 用户中途改方向：使旧方向简报失效，重新确认后只重跑受影响的下游任务。
