# 降智输出指南

> 本文档定义了AI项目专家部门在降智输出阶段的标准方法论。

## 一、核心理念

### 什么是"降智"？
"降智"不是降低质量，而是降低理解门槛。将专业复杂的信息转化为任何人都能快速理解的语言，同时保留核心信息的准确性。

### 降智三原则

| 原则 | 说明 | 示例 |
|------|------|------|
| **说人话** | 禁止使用未经解释的专业术语 | ❌ "该项目采用微服务架构" → ✅ "把一个大系统拆成很多小系统，各管各的" |
| **用比喻** | 每个复杂概念配一个生活化类比 | API = 餐厅服务员，负责把你的点单传给厨房 |
| **给结论** | 先说结论，再说原因 | ❌ "因为A、B、C原因，所以推荐X" → ✅ "推荐X。原因是A、B、C" |

## 二、降智翻译技巧

### 2.1 术语翻译法

**步骤**：
1. 识别专业术语
2. 用大白话解释其本质功能
3. 配一个生活化的类比
4. 说明为什么重要

**模板**：
```
{术语} = {大白话解释}
就像{生活类比}。
它之所以重要，是因为{重要性}。
```

**示例**：
```
Docker = 把软件和它需要的一切打包在一起，放到哪里都能运行
就像集装箱，不管里面装的是什么，外面的接口都是一样的
它解决了"在我电脑上能跑"的问题
```

### 2.2 数字具象化法

将抽象数字转化为可感知的具体描述：

| 抽象 | 具象 |
|------|------|
| "延迟10ms" | "比你眨一次眼还快" |
| "日活100万" | "相当于整个旧金山市的人口" |
| "数据量1TB" | "可以存25万张高清照片" |
| "学习周期3个月" | "大约一个学期的时间" |

### 2.3 结构降维法

将复杂结构简化为可理解的层次：

```
原始结构（复杂）          降智结构（简单）
┌─────────────┐         ┌──────────┐
│ 系统架构     │    →    │ 三大部分  │
│ - 前端       │         │ 1. 用户看到的部分 │
│ - 后端       │         │ 2. 背后处理的部分 │
│ - 数据库     │         │ 3. 存数据的地方   │
│ - 缓存       │         └──────────┘
│ - 消息队列   │
│ - CDN       │
└─────────────┘
```

### 2.4 对比解释法

用用户已知的事物来解释未知的事物：

```
你想了解的 {新概念}，其实就相当于 {已知概念} 在 {新领域} 的应用。

比如：
- 游戏引擎 = 游戏界的"乐高积木套装"
- BIM = 建筑界的"数字孪生"
- RTOS = 硬件界的"专车司机"（普通OS是"公交系统"）
```

## 三、输出格式规范

### 3.1 三层输出结构

#### ⚡ 30秒版（电梯演讲）
- 严格控制在3句话以内
- 第一句：项目是什么
- 第二句：核心价值
- 第三句：适合谁

#### 📖 3分钟版（速读卡片）
- 结构化展示核心信息
- 每个概念配类比
- 包含难度评估和行动建议

#### 🔍 深入版（可选）
- 分模块的通俗解释
- 适合想要进一步了解的用户
- 保持通俗风格但增加细节

### 3.2 排版规范

| 规范 | 说明 |
|------|------|
| Emoji使用 | 适当使用Emoji增强可读性，但不过度 |
| 段落长度 | 每段不超过3-4行 |
| 列表使用 | 多用列表，少用长段落 |
| 重点标注 | 关键信息用**加粗**标注 |
| 类比格式 | 类比统一用"就像……"的格式 |

## 四、质量检查清单

### 可理解性检查
- [ ] 一个完全没有背景知识的人能看懂吗？
- [ ] 是否有未经解释的专业术语？
- [ ] 每个复杂概念是否有类比？

### 信息完整性检查
- [ ] 是否保留了原始分析的核心结论？
- [ ] 是否遗漏了重要的风险提示？
- [ ] 行动建议是否具体可执行？

### 阅读体验检查
- [ ] 能在3分钟内读完3分钟版吗？
- [ ] 排版是否清晰舒适？
- [ ] 语气是否友好不居高临下？

### 行动导向检查
- [ ] 读者读完后知道下一步该做什么吗？
- [ ] 是否有明确的"快速行动指南"？
- [ ] 难度评估是否帮助读者建立合理预期？

## 五、常见错误

### ❌ 错误1：假降智
表面通俗，实际仍然充满术语。
```
错误示例："这个项目用K8s编排容器化微服务，通过CI/CD实现DevOps。"
正确做法：每个术语都要解释，或者干脆用大白话替换。
```

### ❌ 错误2：过度简化
简化到丢失核心信息。
```
错误示例："这就是一个网站。"
正确做法：保留关键特征，"这是一个让用户可以{核心功能}的网站，特别之处在于{独特之处}。"
```

### ❌ 错误3：类比不当
类比与原概念差距过大，反而造成误解。
```
错误示例："微服务就像炒菜，每道菜单独炒。"（差距太大）
正确做法："微服务就像一个餐厅的后厨，每个厨师负责一道菜，各自独立但通过传菜口协作。"
```

### ❌ 错误4：信息过载
试图在降智版中塞入太多信息。
```
错误示例：3分钟版写了3000字。
正确做法：3分钟版控制在800-1200字，详细内容放深入版。
```
