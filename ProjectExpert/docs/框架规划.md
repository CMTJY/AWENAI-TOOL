## 一、设计原则

### 1.1 核心原则
1. **一次只问一个问题** — 引导式提问，避免用户认知过载
2. **职责不重叠** — 每个部门/智能体有明确的职责边界
3. **部门平级协作** — 各部门之间是平级协作关系，不存在从属关系
4. **质量门禁必检** — 每个阶段完成后必须通过质量检查

### 1.2 设计依据
- 用户需求：对话引导式交互 + 全行业覆盖 + 完整输出
- 使用场景：综合场景（内部决策 + 对外输出）
- 当前状态：完全从零开始

## 四、协作流程设计

### 4.1 主流程

```
用户输入
   │
   ▼
┌─────────────────────────────────────────────┐
│  总指挥 (director-main)                      │
│  判断用户意图清晰度，调度对应部门              │
└──────────┬──────────────────────────────────┘
           │
           ▼
┌─────────────────────────────────────────────┐
│  ① 需求引导部 (guidance)                     │
│  意图探针 → 轮询提问（一个一个来）             │
│       → 项目定位师 → 锁定目标项目             │
│  【质量门禁】用户确认项目定位正确              │
└──────────┬──────────────────────────────────┘
           │
           ▼
┌─────────────────────────────────────────────┐
│  ② 情报检索部 (intelligence)                 │
│  全域侦察兵 → 跨行业全面检索                  │
│       → 知识图谱师 → 结构化整理              │
│  【质量门禁】信息覆盖度 ≥ 80%                │
└──────────┬──────────────────────────────────┘
           │
           ▼
┌─────────────────────────────────────────────┐
│  ③ 规划分析部 (planning)                     │
│  技术架构师 → 技术栈 + 知识库 + 学习计划      │
│       → 需求规划师 → 需求分析 + 推进方案      │
│  【质量门禁】计划可执行性验证通过              │
└──────────┬──────────────────────────────────┘
           │
           ▼
┌─────────────────────────────────────────────┐
│  ④ 降智输出部 (simplifier)                   │
│  降智翻译官 → 关键信息降智总结                │
└──────────┬──────────────────────────────────┘
           │
           ▼
┌─────────────────────────────────────────────┐
│  输出总控 (director-output)                   │
│  整合所有成果 → 生成最终交付物                │
└─────────────────────────────────────────────┘
```

### 4.2 质量门禁

| 阶段 | 门禁标准 | 不通过处理 |
|------|----------|-----------|
| 需求引导 | 用户确认项目定位正确 | 继续提问或调整定位 |
| 情报检索 | 核心信息覆盖度 ≥ 80% | 补充检索 |
| 规划分析 | 计划具备可执行性 | 深化分析 |
| 降智输出 | 非技术人员能理解核心内容 | 重新降智 |
