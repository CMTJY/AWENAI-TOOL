# skill-creator：自定义 skill 创建工厂

> 详细格式规范（目录结构、字段约束、命名规则、示例等）请参考：
> [references/reference.md](./references/reference.md)

## 使用场景

本技能支持两种调用场景：

### 场景一：新建自定义 skill
当用户想要创建一个全新的自定义 skill，用于实现特定功能时调用。

### 场景二：将对话封装为 skill
当用户想要将当前的聊天对话修改或封装成一个可复用的 skill 时调用。

---

## 流程

### 1. 识别使用场景并确认需求
- 首先明确用户的使用场景：**新建自定义 skill** 还是 **将对话封装为 skill**
- 与用户**一次一个问题地**交流，了解用户的具体需求
- 无论哪种场景，都需要了解：技能的名称、描述、使用场景、授权类型、兼容性、元数据等信息
- 直到跟用户确认最终完整的需求，才进行下一步。**未确定之前，不开始设计**
- 问问题的形式以**引导性选项或者问题填空的对话框 UI** 的风格

重点关注理解：
- 目的
- 目标用户
- 约束
- 成功标准
- 明确的非目标

### 2. 规划设计思路（强制性）
根据用户需求，规划 **skill 工程的设计思路**，包括但不限于：
- 有无脚本文件
- 需要调用的当前系统配置的 MCP 有哪些
- skill 工程的目录结构
- SKILL.md 文件的内容
- README.md 文件的内容
- references/ 目录下的 reference.md 文件的内容
- assets/ 目录下的静态资源文件的内容
- **再次给用户确认这个设计思路后再进行下一步**

### 3. 实现设计
- 根据设计思路开始实现 skill 工程的设计
- 如果设计思路中**有脚本文件**，则需要在 `scripts/` 目录下创建对应的脚本文件，文件内容要符合 skill 的要求。且需要验证脚本文件通过应用测试，如果没有办法通过测试，停止设计并输出失败结论
- 最终完成设计的地址保存为本次任务的 `skill_storage_path`（见下方"保存路径"约定）
- **SKILL.md 格式检查（强制性）**：创建完成后必须检查以下内容：
  - [ ] 文件开头没有 BOM 字符（UTF-8 字节顺序标记）
  - [ ] frontmatter 格式正确：`---` 开头和结尾，无多余空格
  - [ ] `name` 字段与目录名完全一致
  - [ ] `description` 字段不为空且长度在 1-1024 字符
  - [ ] 正文内容不为空
- **最终验证（强制性）**：
  - [ ] 使用 `Read` 工具读取 SKILL.md 前 10 行，确认 frontmatter 解析正常
  - [ ] 使用 `Glob` 工具列出技能目录下所有文件，确认文件结构完整
  - [ ] 告知用户"技能已创建完成，建议重启应用以加载新技能"
- 最终完成后要重新验证一整个流程，输出测试结果

### 4. 对话封装场景的特殊处理
- 如果用户选择**场景二：将对话封装为 skill**
- 需要从当前聊天记录中提取核心交互模式
- 将对话内容转化为结构化的 skill 指令
- 保留关键的对话上下文和决策逻辑
- 确保封装后的 skill 能够独立运行并复现原始对话的核心功能
- 完成后输出完整的 skill 工程结构供用户确认

---

## 设计优化
- 第一次输出设计后，用户如果需要修改设计的 skill 工程，需要重新询问并明确用户需求，然后根据用户需求重新规划设计思路。
- 在重新规划后的基础上进行修改原设计。

---

## 保存路径（与运行环境解耦）

本 skill **不在文件中硬编码任何本地绝对路径**，避免与具体机器绑定。保存路径按以下优先级确定：

1. **用户在本次对话中明确指定的路径**（最高优先级）：用户说"保存到 X"时，使用 X。
2. **运行环境中已配置的 skill 存储目录**：如果运行平台/宿主应用已提供默认的 skill 存放位置（例如 IDE 的全局 skills 目录），则使用该位置。**不要凭空假设一个固定的本地路径。**
3. **未指定时的处理**：若以上两种来源都没有，则在执行创建动作之前，**必须先向用户询问**保存路径，并提供 1-2 个合理的默认选项（例如"当前仓库的同级 `skills/` 目录"或"宿主应用全局 skills 目录"）。

> 注意：skill 工程本身的源码（SKILL.md、references/、scripts/、assets/）跟随本仓库提交，不需要在每次使用时指定。

## agent-skill 格式规范（精简版）

完整规范请阅读 [references/reference.md](./references/reference.md)。以下是必须遵守的硬性约束：

### 目录结构
```
skill-name/         # 目录名必须与 SKILL.md 中 "name" 字段完全一致
├── SKILL.md        # 必填：存储元数据与技能操作指令
├── scripts/        # 可选：存放可执行代码脚本，该文件夹下不可再新建文件夹
├── references/     # 可选：存放补充说明文档，该文件夹下不可再新建文件夹
└── assets/         # 可选：存放静态资源文件，该文件夹下不可再新建文件夹
```

### SKILL.md frontmatter 必填字段
- `name`：1-64 字符；仅含小写字母(a-z)、数字、连字符(-)；不首尾含连字符、无连续连字符(`--`)；**需与目录名一致**
- `description`：1-1024 字符；说明「技能功能 + 使用场景」，包含关键词

### SKILL.md frontmatter 选填字段
- `license`：简短标注授权类型或引用授权文件
- `compatibility`：1-500 字符；仅当有特殊环境要求时填写（如依赖软件、网络权限、适配产品）
- `metadata`：自定义键值对；**不要在其中放置与具体机器绑定的绝对路径**

### SKILL.md 格式检查清单（创建后必须执行）

| 检查项 | 检查方法 | 通过标准 |
|--------|----------|----------|
| **无 BOM 字符** | 使用 Python 读取文件前 3 个字节 | 不是 `0xEF 0xBB 0xBF` |
| **frontmatter 格式** | 检查文件开头和结尾 | 以 `---\n` 开头，第二个 `---\n` 结束 |
| **name 字段** | 检查 frontmatter 中的 name | 与目录名完全一致，小写字母+数字+连字符 |
| **description 字段** | 检查 frontmatter 中的 description | 不为空，长度 1-1024 字符 |
| **正文内容** | 检查第二个 `---` 之后 | 有非空内容 |

**验证方法：**
```python
# 检查 BOM
with open('SKILL.md', 'rb') as f:
    if f.read(3) == b'\xef\xbb\xbf':
        raise Exception('BOM detected!')

# 检查 frontmatter
with open('SKILL.md', 'r', encoding='utf-8') as f:
    content = f.read()
    if not content.startswith('---\n'):
        raise Exception('Invalid frontmatter start!')
    if '\n---\n' not in content:
        raise Exception('Invalid frontmatter end!')
```

## 补充信息
- 创建 skill 工程和所有文件保存在本次任务确定的 `skill_storage_path`（见"保存路径"约定）。

- 开始**创建前要先一次一个问题地问用户问题**，了解用户创建 skill 的需求，包括技能的名称、描述、使用场景、授权类型、兼容性、元数据等信息。

- 新建的 skill 工程内容及功能，**能够在 SKILL.md 上面体现就在上面体现**。

- 如果 SKILL.md 上**内容过多可以考虑将内容拆分**至 `references/` 目录下的 reference.md 文件，SKILL.md 上只保留核心功能说明，引用路径要清楚，如：[参考文档](./references/reference.md)。

- 创建 skill 功能中，**目标功能要是能不用编写脚本文件就能实现，就不要编写脚本文件**，而是直接在 SKILL.md 上面实现。例如：新建的 skill 需要检索并下载 GitHub 上某主题的项目，先判断当前是否已安装类似的 GitHub MCP 插件；如有，则只需在 SKILL.md 中体现"调用 GitHub MCP 检索并下载符合用户需求的项目"；如无，**暂停创建工作向用户确认是否创建脚本**，再考虑编写脚本实现目标功能。当然，如果用户明确要求某种方式实现，就以用户要为主。

- 创建的 skill 工程，要在 `./skill-name/` 目录下创建 `README.md` 文件，文件内容要包括技能的名称、描述、使用场景、授权类型、兼容性、元数据等信息，帮助用户快速了解创建的 skill。
