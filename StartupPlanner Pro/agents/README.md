## 目录简介

本目录存放 StartupPlanner 创业规划平台的所有智能体规则文件。每个智能体都有独立的配置文件，定义其角色、能力、工作流程和交付物标准。

## 智能体分类说明

智能体按照企业职能划分为六大部门，每个部门负责创业规划的不同维度：

```
┌─────────────────────────────────────────────────────────────┐
│                    StartupPlanner 智能体架构                  │
├───────────────┬─────────────────────────────────────────────┤
│   指挥层      │  _director/                                  │
│               │  • 创业总指挥 - 流程协调与质量把控            │
│               │  • BP总编 - 商业计划书整合输出               │
├───────────────┼─────────────────────────────────────────────┤
│   战略层      │  strategy/                                   │
│               │  • 战略分析师 - 行业分析与商业模式            │
│               │  • 市场调研员 - 用户洞察与市场规模            │
├───────────────┼─────────────────────────────────────────────┤
│   执行层      │  product/     marketing/    finance/        │
│               │  • 产品经理    • 营销策划    • 财务分析师     │
│               │  • MVP设计师   • (增长黑客)   • 融资顾问      │
│               │                • (内容创作)                   │
│               │                • (私域运营)                   │
├───────────────┼─────────────────────────────────────────────┤
│   支撑层      │  organization/   risk/                       │
│               │  • 组织架构师    • 风险管理师                  │
│               │  • 人才规划师                                 │
└───────────────┴─────────────────────────────────────────────┘
```

## 如何使用这些规则

### 1. 规则文件结构

每个智能体规则文件采用统一的 YAML Front Matter + Markdown 格式：

```yaml
---
name: 智能体名称
description: 智能体功能描述
emoji: 表情符号
color: 主题色
---

# 智能体名称

## 身份与记忆
- 角色定义
- 专业背景
- 核心记忆

## 核心使命
- 主要职责
- 关键任务

## 关键规则
- 工作原则
- 约束条件

## 技术交付物
- 标准化输出模板
- 文档格式规范

## 工作流程
- 执行步骤
- 阶段划分

## 沟通风格
- 语言特点
- 表达方式

## 成功指标
- 交付物质量标准
- 完成度检查清单
```

### 2. 调用智能体

#### 通过创业总指挥调用（推荐）

创业总指挥是智能体协作的协调者，能够根据项目阶段自动调度相关部门：

```
用户：我想做一个宠物社交平台

创业总指挥：
1. 进行项目诊断
2. 制定三阶段规划
3. 按阶段调用相应部门
   - 第一阶段：调用战略部
   - 第二阶段：调用产品部、营销部、组织部、财务部
   - 第三阶段：调用风险部、财务部
4. 质量门禁检查
5. 输出完整BP
```

#### 直接调用特定智能体

如果你需要特定领域的专业建议，可以直接调用对应智能体：

```
【调用 财务部-财务分析师】
输入：
- 前序输出：产品部已完成MVP设计，预计首年获客10,000人
- 当前需求：建立首年财务预测模型，包括收入、成本、现金流
- 约束条件：种子轮融资500万，团队10人，SaaS订阅模式

期望输出：
- 3年财务预测表
- 月度现金流预测
- 盈亏平衡分析
- 关键财务指标
```

### 3. 规则文件命名规范

```
{部门目录}/
├── {智能体标识}.md           # 主规则文件
└── (可选) {智能体标识}/      # 子目录（用于复杂智能体）
    ├── README.md
    ├── templates/
    └── examples/
```

命名建议：
- 使用小写字母和连字符（kebab-case）
- 名称应准确反映智能体职能
- 避免使用缩写（除非是通用缩写如mvp、bp）

### 4. 规则文件编写规范

#### YAML Front Matter 必填字段

```yaml
---
name: 智能体中文名称                    # 必填
description: 一句话描述智能体功能        # 必填
emoji: 🎯                               # 可选，用于视觉识别
color: "#FF6B35"                        # 可选，主题色
---
```

#### Markdown 内容规范

1. **身份与记忆**：明确智能体的角色定位和专业背景
2. **核心使命**：列出3-5个主要职责
3. **关键规则**：定义工作原则和约束条件
4. **技术交付物**：提供标准化的输出模板
5. **工作流程**：分步骤说明执行流程
6. **沟通风格**：定义语言特点和表达方式
7. **成功指标**：明确交付物质量标准

### 跨部门数据引用规范

各部门在输出中引用的数据必须保持一致：

1. **市场规模数据** - 以战略部输出为准
2. **用户画像数据** - 以市场调研员输出为准
3. **产品功能清单** - 以产品经理输出为准
4. **财务数据** - 以财务分析师输出为准
5. **团队规模** - 以组织架构师输出为准

### 冲突解决机制

当多部门输出冲突时（如财务模型与营销预算不符）：

1. **创业总指挥介入** - 识别冲突点
2. **数据溯源** - 确认数据来源和假设
3. **协调会议** - 召集相关部门对齐
4. **决策仲裁** - 基于项目目标做出决策
5. **统一输出** - 更新相关部门文档

## 最佳实践

### 对规则编写者

1. **保持角色一致性** - 智能体的专业背景、沟通风格要统一
2. **提供具体模板** - 每个交付物都要有可直接使用的模板
3. **定义明确边界** - 清楚说明智能体的能力边界和不做什么
4. **考虑扩展性** - 规则设计要便于后续扩展和迭代

### 对规则使用者

1. **完整阅读规则** - 了解智能体的能力和限制
2. **提供充分上下文** - 调用时提供足够的前序输出和约束条件
3. **遵循调用规范** - 使用标准的调用格式
4. **检查输出质量** - 对照成功指标检查交付物

## 常见问题

### Q: 可以直接调用智能体而不经过总指挥吗？

A: 可以。如果你需要特定领域的专业建议，可以直接调用对应智能体。但完整的创业规划建议通过总指挥协调，以确保输出的一致性和完整性。

### Q: 智能体的输出可以直接用于融资吗？

A: 智能体输出提供了专业的框架和思路，但建议：
- 结合实际情况调整数据
- 请专业顾问审核关键内容
- 根据投资人反馈迭代优化

### Q: 如何反馈智能体规则的改进建议？

A: 可以通过以下方式：
- 提交Issue描述改进建议
- 提交PR修改规则文件
- 联系维护者讨论

### Q: 营销部的智能体为什么是目录结构？

A: 营销部采用分类目录结构是为了支持更细分的营销职能。每个分类下可以包含多个相关智能体，便于后续扩展和维护。
