# 智能体团队搭建 · 详细规范

## 一、智能体规则文件格式规范

### 1.1 文件结构

每个智能体规则文件遵循以下统一格式：

```markdown
---
name: {中文名称}
description: {一句话描述，说明角色定位和核心职责}
emoji: {emoji图标}
color: {主题色，十六进制}
---

# {中文名称}

## 你的身份与记忆
- **角色**：{角色定位，一句话说明你是谁}
- **个性**：{3-4个性格特征词}
- **记忆**：{你拥有的经验和知识储备}
- **经验**：{你的核心方法论或原则}

## 核心使命

### {职责领域1}
{具体职责描述，3-5条}

### {职责领域2}
{具体职责描述，3-5条}

## 关键规则

### {规则类别1}
{具体规则，包含约束条件和执行标准}

### {规则类别2}
{具体规则，包含约束条件和执行标准}

## 技术交付物

### {交付物1名称}
{交付物模板，使用Markdown格式}

### {交付物2名称}
{交付物模板，使用Markdown格式}

## 工作流程

### 第一步：{步骤名称}
{具体操作说明}

### 第二步：{步骤名称}
{具体操作说明}

### 第N步：{步骤名称}
{具体操作说明}

## 沟通风格
- **场景1**："{示例话术}"
- **场景2**："{示例话术}"

## 成功指标
- {指标1}：{量化标准}
- {指标2}：{量化标准}
```

### 1.2 编写要求

| 要求 | 说明 |
|------|------|
| **身份具体化** | 不要写"你是一个AI助手"，要写具体的角色背景和经验 |
| **职责可执行** | 每个职责都要有明确的输入、输出和判断标准 |
| **规则可验证** | 规则要能被检查是否遵守，避免模糊描述 |
| **模板可直接用** | 交付物模板要填上占位符，用户可以直接使用 |
| **流程有顺序** | 工作流程要按实际执行顺序排列 |
| **风格统一** | 沟通风格要与角色定位匹配 |

## 二、项目目录结构规范

### 2.1 标准结构

```
{ProjectName}/
├── .trae/
│   └── rules/                    # 智能体规则文件
│       ├── _director/            # 总指挥层（可选）
│       │   ├── director-main.md
│       │   └── director-output.md
│       │
│       ├── {department-1}/       # 部门1
│       │   ├── {role-1}.md
│       │   └── {role-2}.md
│       │
│       ├── {department-2}/       # 部门2
│       │   ├── {role-1}.md
│       │   └── {role-2}.md
│       │
│       └── ...                   # 更多部门
│
├── docs/
│   ├── templates/                # 输出模板
│   │   ├── {template-1}.md
│   │   └── {template-2}.md
│   │
│   └── methodology/              # 方法论文档
│       ├── {method-1}.md
│       └── {method-2}.md
│
├── workspace/                    # 工作空间
│   └── {project-instance}/       # 项目实例
│       ├── context/              # 项目上下文
│       ├── drafts/               # 草稿
│       └── output/               # 最终输出
│
└── README.md                     # 项目说明
```

### 2.2 命名规范

| 类型 | 规范 | 示例 |
|------|------|------|
| 项目目录 | PascalCase | `StartupPlanner` |
| 部门目录 | kebab-case | `strategy`, `product` |
| 总指挥目录 | 下划线前缀 | `_director` |
| 规则文件 | kebab-case | `strategy-analyst.md` |
| 模板文件 | kebab-case + 后缀 | `bp-template.md` |
| 方论文档 | kebab-case | `startup-phases.md` |

## 三、常见问题与解决方案

### Q1：部门职责重叠怎么办？

**识别方法**：对比各部门的核心职能，看是否有超过30%的职责重叠。

**解决方案**：
- 重叠部分合并到一个部门
- 或者将重叠职责拆分为更细粒度的子职责，分别归属不同部门

### Q2：用户对框架不满意需要修改？

**处理方式**：
- 不要删除旧版本，而是迭代更新（v1→v2→v3）
- 每次修改后让用户确认
- 记录修改原因，便于回溯

### Q3：GitHub 上没有找到合适的智能体？

**处理方式**：
- 先搜索类似领域的智能体，考虑改造
- 如果确实没有，根据规则文件格式规范自行创建
- 创建时参考已有的高质量智能体（如 MarketMaster 系列）

### Q4：用户已有现成的智能体项目要整合？

**处理方式**：
- 将现有项目作为平级部门纳入架构
- 保留其原有的目录分类结构
- 评估现有智能体覆盖度，标注需要补充的角色
