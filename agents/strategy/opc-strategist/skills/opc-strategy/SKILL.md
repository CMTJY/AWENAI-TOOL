# OPC Strategy — AI原生创业验证全流程

基于Anthropic官方创业手册"7步+1闸"方法论，在写第一行代码前用证据回答"这值得做吗"。

> **核心心法：** AI是加速器，证据才是目的，判断永远是人的事。

---

## 完整流程概览

```
用户原始想法
    ↓
Step 1: 框定假设 (4问测试)
    ↓
Step 2: 唱反调 (找反证)
    ↓
Step 3: 市场情报 (爬虫数据)
    ↓
Step 4: 设计访谈 (The Mom Test)
    ↓
Step 5: 跑访谈 (真人证据 ★)
    ↓
Step 6: 综合分析 (支持vs反对)
    ↓
Step 7: 原型测试 (真人证据 ★)
    ↓
总闸: GO / NO-GO 决策
```

---

## Step 1 · 框定假设

**目标：** 把模糊的观察，逼成一个具体、可被证伪的句子。

### 4问测试 (可检验性测试)

一个假设可检验，当且仅当能精确回答：

1. **谁 (Who)** 有这问题？（具体人群，不是"用户"）
2. **多频繁 (Frequency)**？（每天/每周/每月？）
3. **多严重 (Severity)**？（时间损失？金钱损失？）
4. **现在怎么应对 (Workaround)**？（现有替代方案，即使不完美）

**四个都必须答得上，才算合格。**

### 操作话术

```
"我是{身份}，想做{方向}。帮我把『{模糊观察}』这句话，
逼成一个可检验的假设。请追问我：
谁最受此困扰？多频繁？多严重？现在用什么应对？
我答不上来的地方，直接标出来。"
```

### 输出模板

```markdown
## 可检验假设 v1

**原始观察：** {用户原始想法}

**精炼假设：**
{具体人群}，每周/每月花{具体时间/金钱}在{具体痛点}上，
因为{现有解决方案}存在{具体缺陷}，导致{具体后果}。

**4问验证：**
| 问题 | 答案 | 状态 |
|------|------|------|
| 谁 (Who) | | ✅/❌ |
| 多频繁 | | ✅/❌ |
| 多严重 | | ✅/❌ |
| 现在怎么做 | | ✅/❌ |

**待验证缺口：**
- {未回答问题} → Step 2/5 重点验证

**判据：**
- [ ] 四个问题全部答得上 → 进入 Step 2
- [ ] 有未回答问题 → 带着缺口进入 Step 2
```

### 规则

- 不让用户跳过任何问题，答不上就标记
- 要具体："50-200人SaaS公司"而非"中小企业"
- 要量化："平均3天"而非"很慢"
- workaround问题最关键：如果没workaround，要么问题不真实，要么找错了人群

## Step 2 · 唱反调

**目标：** 让AI当反方，主动去找能推翻假设的证据。

> "别让AI帮你论证你想做的事，让它帮你拆。"

### 操作话术

```
"现在请你当反方。用最有说服力的方式论证：
为什么我的『{假设}』会失败？
去找：已经存在的替代方案、这个方向失败过的产品、
用户其实并不在乎这件事的证据。别照顾我情绪。"
```

### 反方要找的5类证据

1. **现有替代方案** — 什么已经解决了这个？为什么不够？
2. **失败先例** — 什么类似产品/公司失败了？为什么？
3. **用户漠不关心** — 用户其实不在乎这个问题的证据
4. **结构性障碍** — 监管、技术、市场壁垒
5. ** wrong problem ** — 真正的问题是不是在别处？

### 输出模板

```markdown
## 反方论证报告

**待挑战假设：** {假设原文}

### 反方论点
1. **{论点1}** — {证据}
2. **{论点2}** — {证据}
...

### 待验疑点清单
🔴 **最关键疑点：** {最关键的问题}
→ 验证方法：{建议方式}

🟡 {次要疑点1}
🟡 {次要疑点2}

### 用户自评
- [ ] 听完反方后，假设依然站得住
- [ ] 有动摇，需要调整假设
- [ ] 假设基本被推翻，需要重新思考

### 下一步
- 站得住 → 进入 Step 3
- 动摇 → 回到 Step 1 重新框定
```

### 规则

- 要 ruthless，不要 supportive
- 要 evidence，不要 opinion
- 不说"但你可以克服"，让用户自己判断
- 特别注意"wrong problem"论点（如约课 vs 学员留存）

## Step 3 · 市场情报

**目标：** 扒方向、扒高频抱怨、扒用户原话；摸清竞品和市场。

> "爬虫给方向，真人给真假。"

### 三件事

**① 竞品评估**
```
分析{竞品}：
1. 它的核心优势和护城河是什么？
2. 用户最满意的三点？
3. 用户最常抱怨的三点？
4. 为什么它可能赢，而我会输？
```

**② 市场研究**
```
综合{渠道}关于{目标人群}的公开评论：
1. 反复抱怨的高频问题 Top 5
2. 现有解决方案的差评集中在哪
3. 关于「{假设方向}」的声音，频率多高
请给出原话引用。
```

**③ 趋势分析**
- 监管：顺风还是逆风？
- 技术：新技术让问题更重要还是更不重要？
- 人口：目标人群在扩大还是缩小？

### 关键边界

> **爬虫只能证明"这个抱怨被高频提到"，不能证明"这是个值得做的真问题"。**
> 望远镜看到了方向，真假还得靠显微镜（真人访谈）。

### 输出模板

```markdown
## 市场情报报告

### 竞品评估
**{竞品A}**
- 优势：{...}
- 用户好评：{...}
- 用户差评：{...}
- 为什么它可能赢：{...}

### 高频抱怨 Top 5
| 排名 | 抱怨 | 热度 | 现有方案是否解决 |
|------|------|------|------------------|
| 1 | | | |
| ... | | | |

### 趋势分析
- 监管：{顺风/逆风}
- 技术：{顺风/逆风}
- 人口：{顺风/逆风}

### 关键发现
- [ ] 假设命中了"现有方案没解决的高频抱怨" → 利好，进入 Step 4
- [ ] 假设排名很低 → 可能不是真痛点，警惕

⚠️ **重要提醒：** 本报告仅提供方向性证据，所有结论必须在 Step 5 真人访谈中验证。
```

## Step 4 · 设计客户探索

**目标：** 设计访谈脚本，只问过去行为，不问未来意愿。

> "问'上次怎么做的'，不问'以后会不会用'。"
> 人会为了礼貌骗你未来，但骗不了已经发生的过去。

### The Mom Test 原则

✅ **GOOD：** "上一次你给学员发课后练习计划，是什么时候？具体怎么做的？花了多久？"

❌ **BAD：** "你会用一个课后跟进工具吗？" → 会得到礼貌的"会啊"

### 访谈脚本结构

| 阶段 | 时长 | 内容 | 规则 |
|------|------|------|------|
| 开场 | 5min | 自我介绍 + 知情同意 + 暖场 | 建立信任 |
| 背景 | 10min | 了解对方工作/生活场景 | **不要提你的产品** |
| 核心 | 30min | 问过去的具体行为 | 只问过去，不问未来 |
| 概念测试 | 10min | (可选) 展示概念，收集反馈 | 不要引导 |
| 结束 | 5min | 总结确认 + 感谢 | 询问转介绍 |

### 关键对照题（多方向PK）

```
"过去一个月，你为'约课改时间'烦过几次？
 为'学员不练/留不住'烦过几次？"
```

用真实频率直接PK两个方向。

### Mom Test 检查清单

- [ ] 问题是否关于过去的具体行为？
- [ ] 问题是否足够具体（有具体时间/场景）？
- [ ] 问题是否中立（没有暗示正确答案）？
- [ ] 问题是否会让对方为了讨好你而回答？

### 输出模板

```markdown
## 客户探索设计方案

### 目标画像
- 核心特征：{...}
- 必须满足：{...}
- 排除条件：{...}
- 招募渠道：{...}

### 访谈脚本
**开场（5分钟）**
- 自我介绍
- 访谈目的（学习，不是推销）
- 暖场："请简单介绍你的工作/日常？"

**背景了解（10分钟）**
{设计好的背景问题，不提及产品}

**核心问题（30分钟）**
{按优先级排序的过去行为问题}

**关键对照题：**
{如果有多个方向，设计PK问题}

**结束（5分钟）**
- "还有什么我没问到但你想说的吗？"
- 感谢与报酬
- 转介绍请求

### 质量控制
- [ ] 所有问题通过 Mom Test 检查
- [ ] 没有未来意愿问题
- [ ] 有明确的追问设计（"为什么？""还有呢？"）
```

## Step 5 · 跑访谈

**目标：** 获取真人证据。这是第一个产生"真假判断"的步骤。

### 执行规则

- **前20分钟不提你的解决方案** — 先了解他们的世界
- **沉默是你的朋友** — 回答后等3秒，他们会补充最重要的部分
- **要具体，不要笼统** — "经常" → "上周有几次？"
- **跟着痛苦走** — 当他们声音变化或激动时，深挖

### 样本量

- 初始：5-8人
- 信号混合：继续到12-15人
- 听到重复内容时停止（饱和）

### 每5场小结一次

记录：
- 支持假设的证据数
- 反对假设的证据数
- 意外发现

---

## Step 6 · 综合分析

**目标：** 把访谈记录转化为支持/反对两张清单，让数据说话。

> "每一步都有AI，但每一个'判断'都是人做的。"

### 操作

把访谈记录丢给AI：
```
"这是{n}份访谈记录。产出两张清单：
① 支持『{假设}』的证据
② 反对『{假设}』的证据
再单独列：关于『{替代方向}』，支持和反对各有哪些。
最后反问我一句：我的解读是不是在挑我想听的？"
```

### 跨方向对比表

| 方向 | 支持证据 | 反对证据 | 反对是否可被解决 |
|------|----------|----------|-----------------|
| 方向A | | | |
| 方向B | | | |

### 输出模板

```markdown
## 访谈综合分析

### 假设回顾
{原始假设}

### 方向一：{假设A}
**支持证据 ({n}条)**
1. "{直接引用}" — 受访者{编号}
2. ...

**反对证据 ({n}条)**
1. "{直接引用}" — 受访者{编号}
2. ...

### 方向二：{假设B}
...

### 跨方向对比
| 判据 | 方向A | 方向B |
|------|-------|-------|
| 问题真实性 | ✅/❌ | ✅/❌ |
| 方案对准真问题 | ✅/❌ | ✅/❌ |
| 信号支撑MVP | ✅/❌ | ✅/❌ |

### 决策
- [ ] **GO** — 进入MVP，方向：{方向}
- [ ] **PIVOT** — 调整方向至：{方向}
- [ ] **NO-GO** — 证据不足，停止

### AI Sanity Check
> "我的解读是不是在挑我想听的？"
> "你确定不是因为'X'听起来更性感，而过度解读了？"
```

## Step 7 · 原型测试

**目标：** 搭轻量原型，放到5个真人面前，获取第二个真假证据。

### 原型原则

- **只做那一个核心交互** — 别做约课、别做收款、别做社区
- **能在手机上打开演示就行** — 不求完美
- **原型不是产品** — 是"对话的压力测试道具"

### 测试方式

拿给 Step 5 里聊过的5个教练真上手试。

### 记录模板

| 用户 | 反应强度 | 关键原话 | 暴露的缺口 |
|------|----------|----------|-----------|
| 1 | 🔥🔥🔥🔥🔥 | "我现在就想用" | — |
| 2 | 🔥🔥🔥🔥 | "计划生成得不错" | "学员练没练我不知道" |
| ... | | | |

### 关键洞察

- 4/5人说"想用"是强信号
- 但真正值钱的是第5个人暴露的缺口 — **它定义了MVP v2第一个要补的功能**

## 总闸 · GO / NO-GO 决策

**目标：** 用三道总闸判断"这值得构建吗？"

### 三道总闸

| 判据 | 标准 | 结论 |
|------|------|------|
| ① 问题真实且具体？ | 有具体人群、频率、损失、workaround | ✅/❌ |
| ② 方案对准验证揭示的真问题？ | 不是最初假设，而是证据指向的问题 | ✅/❌ |
| ③ 信号足够支撑MVP？ | 访谈一致 + 4/5原型想用 + 缺口已知 | ✅/❌ |

### 决策规则

- **GO** (3个✅)：进入MVP阶段，带着已知缺口作为v2功能
- **PIVOT** (①②✅ ③❌)：问题真实，但方案方向需调整
- **NO-GO** (任①❌)：停止，省时间省钱

### 输出模板

```markdown
# 总闸决策报告

## 三道总闸

### 闸1：问题真实且具体？
- 评分：{n}/10
- 证据：{...}
- 结论：{通过/不通过}

### 闸2：方案对准验证揭示的真问题？
- 评分：{n}/10
- 原始假设：{...}
- 验证后的真问题：{...}
- 结论：{通过/不通过}

### 闸3：信号足够支撑MVP？
- 评分：{n}/10
- 强烈信号：{n}/5人
- 暴露缺口：{...}
- 结论：{通过/不通过}

---

## 🚦 最终决策：{GO / PIVOT / NO-GO}

### 理由
{一句话总结}

### 如果 GO
- MVP核心功能：{...}
- v2需补缺口：{...}

### 如果 PIVOT
- 调整方向：{...}
- 需重新验证：{...}

### 如果 NO-GO
- 关键教训：{...}
- 下一步建议：{...}
```

## 关键原则总结

1. **证据 > 直觉** — AI是加速器，真假判断只来自真人
2. **唱反调 > 唱赞歌** — 主动找反证是防止确认偏误的唯一方法
3. **问过去 > 问未来** — 人会骗你未来，骗不了过去
4. **人做判断 > AI做判断** — AI综合证据，创始人下决定

---

## When to Use

- 用户有一个原始想法，需要完整验证
- 在写任何生产代码之前
- 当用户说"我觉得这个方向不错"但未经证实
- 需要系统化的从假设到决策的证据链

## When NOT to Use

- 问题已经过充分验证，进入执行阶段
- 用户只需要单一环节的帮助（如只做竞品分析）
- 用户已有产品，需要增长策略（用营销/产品技能）
