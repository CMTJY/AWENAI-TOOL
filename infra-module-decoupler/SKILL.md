# infra-module-decoupler — 基础设施模块解耦器

## 目标

帮助用户将项目中指定的目标模块（如日志、存储、缓存、驱动等）改造为**纯基础设施层模块**，即：
- ✅ 只能被其他模块依赖
- ✅ 不依赖任何非系统/非标准库的业务文件
- ✅ 每个模块可独立编译、提取、复用到其他项目

---

## 触发条件

当用户说类似以下内容时激活本技能：
- "检查XX模块的依赖关系，把它解耦成独立模块"
- "分析这几个文件的依赖，让它们只能被依赖不能依赖别人"
- "把日志、存储模块封装成可独立使用的模块"
- "分析违规依赖，给出修改方案"

---

## 执行流程总览

```
Phase 1: 依赖分析     →  读取模块，全项目搜索引用关系
Phase 2: 违规识别     →  区分系统依赖 vs 业务依赖，定位违规点
Phase 3: 方案设计     →  为每个违规点匹配解耦策略，生成修改方案报告
Phase 4: 逐步执行     →  逐点修改代码，每次修改后验证，最终全局验证
```

---

## Phase 1：依赖分析

### 1.1 创建任务列表
使用 TodoWrite 创建任务规划，包含：
- 读取目标模块（所有 `.h` 和 `.c` 文件）
- 搜索目标模块被哪些文件引用（被依赖方）
- 搜索目标模块引用了哪些非系统文件（违规依赖方）

### 1.2 读取目标模块
用 Read 工具读取用户指定的所有模块文件（头文件 + 实现文件）。

### 1.3 搜索引用关系
对每个目标模块执行两类搜索：

**搜索A：谁依赖我（被依赖方）**
```
Grep: '#include "模块头文件名"'  →  全项目范围
```
输出：依赖该模块的所有文件列表。

**搜索B：我依赖谁（依赖方）——聚焦非系统文件**
```
逐行检查模块的 #include 指令：
- ✅ 系统文件：<...> 或 esp_xxx.h / nvs.h / nvs_flash.h / esp_spiffs.h / esp_err.h / stdbool.h / stdint.h / string.h / time.h / stdarg.h / stdio.h / sys/stat.h / unistd.h 等
- ❌ 业务文件："xxx.h"（如 "plant_config.h"、 "oled_display.h"、"hx_algo.h"、"logger.h" 等其他模块的路径）
```

### 1.4 输出依赖关系表
以表格形式展示（与本次对话中的格式一致）：
- 模块被引用统计表
- 模块内部依赖检查表（每行标注 ✅合规 / ❌违规）
- 汇总结论：违规总数

---

## Phase 2：违规识别

### 2.1 深入分析每个违规点
对每个违规依赖，搜索：
- 违规代码的具体位置（行号）
- 违规代码的调用情况（是否是死代码？）

### 2.2 死代码判断标准
```
Grep 搜索函数名 → 全项目范围
结果 = 0次外部调用  →  死代码（可直接删除）
结果 > 0次外部调用  →  活代码（需要策略处理）
```

### 2.3 深层依赖链分析
如果违规依赖的文件自身还依赖其他文件（如 `hx_algo.h` → `temperature_algo.h` → `humidity_algo.h` → `watering_algo.h`），需要追踪完整依赖链，评估实际影响深度。

### 2.4 输出违规详情
每个违规点输出：
- 位置（文件和行号，使用 clickable file link）
- 依赖原因
- 外部调用情况表
- 影响评估

---

## Phase 3：方案设计

### 3.1 解耦策略选择

为每个违规点匹配策略，详见 [解耦策略参考](./references/strategies.md)。

| 策略 | 使用场景 | 操作 |
|------|---------|------|
| **A：死代码删除** | 违规代码无外部调用方 | 直接删除相关 include + 变量 + 函数 |
| **B：系统API替换** | 违规依赖可被同功能系统API替代 | 替换宏/函数调用为系统等价物（如 `LOGI` → `ESP_LOGI`） |
| **C：业务逻辑归还** | 违规代码属于业务层职责，不应在基础设施层 | 从目标模块删除，由业务调用方用系统原语自行实现 |

### 3.2 设计核心原则
1. **基础设施层不包含业务类型** — `plant_type_t`、`hx_data_t` 等业务枚举/结构体不属于基础设施模块
2. **不迁移代码** — 不要把一个模块的业务代码搬到另一个模块，而是删除后由调用方用底层 API 自行实现
3. **优先减法** — 宁可删代码，不加间接层（回调、中间层等），除非确实有必要
4. **纯删除优先** — 如果是死代码，直接删除；如果可以替换为系统API，直接替换；如果代码属于业务层，回归业务模块

### 3.3 输出完整报告
包含：
1. 每个违规点的具体修改方案（代码级别）
2. 受影响的外部调用方及改动建议
3. 修改影响范围汇总表
4. 修改前后依赖关系图对比
5. 死代码清理清单

### 3.4 用户确认
**必须**让用户确认方案后再进入 Phase 4 执行。

## Phase 4：逐步执行

### 4.1 执行原则
- 一次只修改一个违规点
- 修改后立即验证
- 使用 SearchReplace 精确修改，避免重写整个文件
- 如果文件被意外损坏，用 `git checkout` 恢复后重试

### 4.2 每种策略的执行步骤

**策略A（死代码删除）：**
1. SearchReplace 删除 `.c` 中的 `#include "xxx.h"`
2. SearchReplace 删除相关 static 变量
3. SearchReplace 删除相关函数
4. SearchReplace 删除调用代码块
5. SearchReplace 删除 `.h` 中的函数声明

**策略B（系统API替换）：**
1. SearchReplace 将 `#include "logger.h"` 替换为 `#include "esp_log.h"`
2. 用 Grep 统计所有需要替换的宏调用次数
3. 逐批 SearchReplace：`LOGI(` → `ESP_LOGI(`、`LOGW(` → `ESP_LOGW(`、`LOGE(` → `ESP_LOGE(`、`LOGD(` → `ESP_LOGD(`
4. 确认无遗漏

**策略C（业务逻辑归还）：**
1. SearchReplace 删除 `.h` 中的违规 include 和函数声明
2. SearchReplace 删除 `.c` 中函数实现（大段删除）
3. 确认调用方列表
4. **不迁移代码** — 仅告知调用方需要用哪些系统原语替代

> 策略C不要求在本次执行中修改调用方，因为调用方在不同的业务模块中，属于另一个变更范围。但必须在执行报告中标注调用方待修改清单。

### 4.3 每步验证
每次修改后：
```
✅ Grep 确认违规关键词已消失
✅ Grep 确认 include 列表合规
✅ 文件行数变化合理
```

### 4.4 最终全局验证
全部修改完成后：
1. 对所有模块执行 Grep 确认无残留违规关键词
2. 列出每个模块的最终 include 依赖清单
3. 确认所有非系统文件依赖已清除
4. 输出完整的依赖关系图

## 注意事项

1. **不做假设**：删除前必须搜索确认代码是否真的无人调用
2. **保护现有功能**：只用等价的系统API替换，不改功能逻辑
3. **增量修改**：优先 SearchReplace，避免重写整个文件
4. **可逆操作**：如文件损坏，及时 `git checkout` 恢复
