# skill-creator：自定义skill创建工厂

## 使用场景

本技能支持两种调用场景：

### 场景一：新建自定义skill
当用户想要创建一个全新的自定义skill，用于实现特定功能时调用。

### 场景二：将对话封装为skill
当用户想要将当前的聊天对话修改或封装成一个可复用的skill时调用。

---

## 流程

### 1️. 识别使用场景并确认需求
- 首先明确用户的使用场景：**新建自定义skill** 还是 **将对话封装为skill**
- 与用户**一次一个问题地**交流，了解用户的具体需求
- 无论哪种场景，都需要了解：技能的名称、描述、使用场景、授权类型、兼容性、元数据等信息
- 直到跟用户确认最终完整的需求，才进行下一步。**未确定之前，不开始设计**
- 问问题的形式以**引导性选项或者问题填空的对话框UI**的风格

重点关注理解：
- 目的
- 目标用户
- 约束
- 成功标准
- 明确的非目标
---

### 2.规划设计思路（强制性）
根据用户需求，规划**skill工程的设计思路**，包括但不限于：
- 有无脚本文件
- 需要调用的当前系统配置的MCP有哪些
- skill工程的目录结构 
- SKILL.md文件的内容
- README.md文件的内容，
- references/目录下的reference.md文件的内容
- assets/目录下的静态资源文件的内容。
- **再次给用户确认这个设计思路后再进行下一步**
---

### 3.实现设计
- 根据设计思路开始实现skill工程的设计
- 如果设计思路中**有脚本文件**，则需要在scripts/目录下创建对应的脚本文件，文件内容要符合skill的要求。且需要验证脚本文件通过应用测试，如果没有办法通过测试，停止设计并输出失败结论
- 最终完成设计的地址保存在`metadata.skill_storage_path`
- **SKILL.md 格式检查（强制性）**：创建完成后必须检查以下内容：
  - [ ] 文件开头没有 BOM 字符（UTF-8 字节顺序标记）
  - [ ] frontmatter 格式正确：`---` 开头和结尾，无多余空格
  - [ ] `name` 字段与目录名完全一致
  - [ ] `description` 字段不为空且长度在 1-1024 字符
  - [ ] 正文内容不为空
- **最终验证（强制性）**：
  - [ ] 使用 `Read` 工具读取 SKILL.md 前 10 行，确认 frontmatter 解析正常
  - [ ] 使用 `Glob` 工具列出技能目录下所有文件，确认文件结构完整
  - [ ] 告知用户"技能已创建完成，建议重启应用以加载新技能"
- 最终完成后要重新验证一整个流程，输出测试结果

### 4.对话封装场景的特殊处理
- 如果用户选择**场景二：将对话封装为skill**
- 需要从当前聊天记录中提取核心交互模式
- 将对话内容转化为结构化的skill指令
- 保留关键的对话上下文和决策逻辑
- 确保封装后的skill能够独立运行并复现原始对话的核心功能
- 完成后输出完整的skill工程结构供用户确认

---

## 设计优化
- 第一次输出设计后，用户如果需要修改设计的skill工程，需要重新询问并明确用户需求，然后根据用户需求重新规划设计思路。
- 在重新规划后的基础上进行修改原设计

## agent-skill的格式规范
- 一个有效技能以独立目录形式存在，最小结构仅需包含必填文件 SKILL.md，可根据功能需求补充3个可选目录，完整结构如下：

```
skill-name/      # 目录名必须与 SKILL.md 中 "name" 字段完全一致
├── SKILL.md     # 必填：存储元数据与技能操作指令
├── scripts/     # 可选：存放可执行代码脚本，该文件夹下不可再新建文件夹
├── references/  # 可选：存放补充说明文档，该文件夹下不可再新建文件夹
└── assets/      # 可选：存放静态资源文件，该文件夹下不可再新建文件夹
```
- SKILL.md 核心格式：
```
---
name: skill-name  #必填，1-64字符；仅含小写字母(a-z)、数字、连字符(-);不首尾含连字符、无连续连字符(--);需与目录名一致

description: 必填，1-1024字符；需说明「技能功能+使用场景」,包含关键词(方便智能体识别)。

license: 选填，简短标注授权类型或授权文件引用，例：license:Apache-2.0/license: 详见LICENSE.txt

compatibility:选填，1-500字符；仅当技能有特殊环境要求时填写(如依赖软件、网络权限、适配产品)，否则无需填写。

metadata:选填，自定义键值对，键名建议唯一(避免冲突),可存储作者、版本、依赖等信息。
---
Markdown正文(技能指令)
```

### SKILL.md 格式检查清单（创建后必须执行）

在创建完 SKILL.md 后，必须按照以下清单进行检查：

| 检查项 | 检查方法 | 通过标准 |
|--------|----------|----------|
| **无 BOM 字符** | 使用 Python 读取文件前3个字节 | 不是 `0xEF 0xBB 0xBF` |
| **frontmatter 格式** | 检查文件开头和结尾 | 以 `---\n` 开头，第二个 `---\n` 结束 |
| **name 字段** | 检查 frontmatter 中的 name | 与目录名完全一致，小写字母+数字+连字符 |
| **description 字段** | 检查 frontmatter 中的 description | 不为空，长度 1-1024 字符 |
| **正文内容** | 检查第二个 `---` 之后 | 有非空内容 |

**验证方法**：
```python
# 检查 BOM
with open('SKILL.md', 'rb') as f:
    if f.read(3) == b'\xef\xbb\xbf':
        raise Exception('BOM detected!')

# 检查 frontmatter
with open('SKILL.md', 'r', encoding='utf-8') as f:
    content = f.read()
    if not content.startswith('---\n'):
        raise Exception('Invalid frontmatter start!')
    if '\n---\n' not in content:
        raise Exception('Invalid frontmatter end!')
```
正文无强制格式限制，核心作用是为智能体提供清晰的操作指引，建议结构化呈现以下内容：
 -分步执行说明(按操作逻辑排序);
 -输入/输出示例(明确预期格式);
 -常见边界情况与异常处理方案。
约束要求：建议正文容量≤5000 tokens(约500行),超长内容需拆分至references/目录。如详细规范见./skill-name/references/目录下的reference.md文件
```

## 补充信息
- 创建skill工程和所有文件保存在`metadata.skill_storage_path`

- 开始**创建前要先一次一个问题地问用户问题**，了解用户创建skill的需求，包括技能的名称、描述、使用场景、授权类型、兼容性、元数据等信息。

- 新建的skill工程内容及功能，**能够在SKILL.md上面体现就在上面体现**

- 如果SKILL.md上**内容过多可以考虑将内容拆分**至references/目录下的reference.md文件，SKILL.md上只保留核心功能说明，引用路径要清楚，如：[参考文档](./references/reference.md)。

- 创建skill功能中，**目标功能要是能不用编写脚本文件就能实现，就不要编写脚本文件**，而是直接在SKILL.md上面实现。例子：新建的skill需要具备以下功能：检索并下载GitHub的关于某某主题的项目，要先判断当前有没有安装类似的GitHub MCP插件，如果有，只需要在SKILL.md上面体现"调用GitHub MCP检索并下载符合用户需求的项目"，如果没有，**暂停创建工作向用户确认是否创建脚本，再考虑编写脚本实现目标功能**。当然，如果用户明确要求某种方式实现，就以用户要为主

- 创建的skill工程，要在./skill-name/目录下创建README.md文件，文件内容要包括技能的名称、描述、使用场景、授权类型、兼容性、元数据等信息.帮助用户快速了解创建的skill
